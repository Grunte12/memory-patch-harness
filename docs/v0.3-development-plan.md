# v0.3 Development Plan

This branch starts the development version after the v0.2 first public push.

The goal is not to copy `claude-obsidian`. The goal is to understand its architecture, decide which problems it solves at a different layer, and improve Memory Patch Harness without losing its smaller contract-first shape.

## Architecture Studied: claude-obsidian

`claude-obsidian` is best understood as an Obsidian vault operating system for Claude Code. It ships a working vault, skills, commands, scripts, tests, transport detection, retrieval, concurrency controls, and methodology routing.

Its architecture has several layers:

```text
source files / raw inputs
  -> ingest skills
  -> wiki pages: sources, entities, concepts, sessions
  -> index + hot cache + log
  -> query/retrieve/lint/save/autoresearch skills
  -> optional Obsidian CLI / MCP / filesystem transport
```

Key architectural components:

| Component | Role in claude-obsidian | What it teaches us |
|---|---|---|
| Hot cache (`wiki/hot.md`) | Short recent-context startup surface | Repeated agent sessions need a small, refreshable context view separate from canonical notes |
| Index (`wiki/index.md`) | Master navigation map | A human-readable source map is useful before deep retrieval |
| Log (`wiki/log.md`) | Chronological activity trail | Chronology is useful as evidence, but should not be the only memory layer |
| Ingest/query/save/lint skills | Separate operations over the vault | Split memory work by operation, not by one giant prompt |
| Transport detection | Chooses Obsidian CLI, MCP, or filesystem | Tool choice should be detected and recorded rather than guessed every run |
| Hybrid retrieval | BM25 plus optional contextual prefix/rerank | Retrieval should be optional, measurable, and degrade gracefully |
| Methodology router | Generic, LYT, PARA, Zettelkasten filing modes | Organization should be configurable and not hardcoded |
| Multi-writer lock | Prevents parallel write corruption | If multiple agents can write, locking or single-writer discipline is necessary |
| Lint skill | Finds orphans, dead links, gaps, stale claims | Memory needs maintenance, not only writes |
| Audit docs/tests | Records architectural claims and failure fixes | Public claims need repeatable tests and visible audit history |

## Difference From Memory Patch Harness

Memory Patch Harness is intentionally smaller.

```text
verified task evidence
  -> lead agent writes Memory Patch
  -> Memory Curator validates/routes/links
  -> canonical Markdown memory
  -> Brain Brief recall
  -> optional Derived Index / Hot Context Pack
```

The key distinction:

- `claude-obsidian` is a full vault runtime.
- `memory-patch-harness` is a portable memory contract and eval harness that can sit inside many agent/vault systems.

## Architecture Lessons To Adopt

### 1. Hot context should be generated, not hand-maintained

Our `Hot Context Pack` matches the same need as `wiki/hot.md`, but ours should stay contract-based and generated from canonical notes.

v0.3 direction:

- keep `hot-context-pack` as derived, not canonical;
- add a demo that renders it into Markdown;
- document when it should be invalidated.

### 2. Memory needs a map before retrieval gets fancy

`claude-obsidian` relies on `index.md` before deeper reads. This supports our `Derived Index` direction.

v0.3 direction:

- create a tiny demo memory workspace with a source map;
- keep derived indexes generic enough for filesystem, Graphify, BM25, or manual maps;
- do not add vector DB by default.

### 3. Tool/transport choice should be explicit

`claude-obsidian` records transport state in `.vault-meta/transport.json`.

v0.3 direction:

- do not implement Obsidian transport yet;
- add adapter guidance that each platform should record its memory access method: filesystem, MCP, Obsidian REST, Obsidian CLI, or custom;
- keep core scripts filesystem-only for now.

### 4. Methodology support should be optional

PARA, LYT, and Zettelkasten are useful, but forcing one would make this harness too opinionated.

v0.3 direction:

- keep `generic` as default;
- add optional note-layout examples, not a mandatory router;
- let Memory Curator report when a vault convention exists and follow it.

### 5. Write safety matters only when writes become automated

`claude-obsidian` needs locks because parallel agents write vault pages.

v0.3 direction:

- keep current harness read/validate/eval focused;
- if we add an apply-to-vault command, add a simple lock or single-writer gate at the same time;
- do not add locking before there is an automated write path.

### 6. Lint is the missing maintenance lane

The most useful architectural idea to borrow is not a feature, but a maintenance loop.

v0.3 direction:

- add a memory lint command or eval fixture for:
  - duplicate claims,
  - missing provenance,
  - stale lifecycle triggers,
  - derived outputs promoted as truth,
  - notes with no incoming links,
  - vague claims.

### 7. Live evaluation matters more than more architecture

`claude-obsidian` documents benchmark/audit history. Our current public gap is live-model results.

v0.3 direction:

- run at least one live-model comparison;
- publish results in `docs/live-model-results.md`;
- keep proxy evals separate from live claims.

## What We Should Not Adopt Yet

- Full Obsidian vault plugin scope.
- Bundled images, GIFs, or heavy demo media.
- Forced methodology routing.
- Default external API retrieval.
- Multi-writer lock without an automated writer.
- Large seed wiki that makes the harness look bigger than it is.

## v0.3 Candidate Milestones

| Priority | Milestone | Why |
|---|---|---|
| P0 | Live-model benchmark result | Converts architecture into evidence |
| P0 | Demo memory workspace | Makes the workflow understandable in five minutes |
| P1 | Memory lint command | Adds maintenance loop without heavy infrastructure |
| P1 | Rendered Hot Context demo | Shows cache-style recall without becoming CAG infrastructure |
| P1 | Adapter transport guidance | Helps OpenCode, Claude Code, and generic agents connect safely |
| P2 | Optional methodology templates | Supports PARA/LYT/Zettelkasten-style users without forcing them |
| P2 | Write lock helper | Only if an apply-to-vault command is added |

## Definition Of Done For v0.3

- `npm run check` passes.
- No private vault or transcript content is committed.
- Architecture docs clearly say this is a contract harness, not a full Obsidian runtime.
- Live-model benchmark status is honest.
- New features remain optional and contract-backed.
- License and attribution files remain accurate.
